Техническое задание на проект «Автоматизированная кассовая система кальянного бара»
1. Название проекта

«Автоматизированная кассовая система кальянного бара».

2. Цель и задачи

Цель проекта — разработать программный продукт на языке Java для автоматизации кассового обслуживания в кальянном баре. Система должна упростить процесс оформления заказов, учёта кальянов и вкусов, а также обеспечить удобное обслуживание VIP-клиентов и ведение истории продаж.

Задачи проекта:

Провести анализ требований и составить архитектуру будущей системы.

Реализовать основные функции системы: управление ассортиментом (кальяны и вкусы), оформление заказов, работа с меню, VIP-бронирование, сохранение истории заказов, система авторизации администратора.

Создать первоначальную консольную версию приложения (этап 1), реализовав логику работы системы и обеспечив хранение данных в памяти или файлах.

Разработать графический интерфейс с тёмной темой (этап 2) для повышения удобства работы с программой.

Обеспечить интеграцию системы с базой данных на заключительном этапе (этап 3, детали БД разрабатываются позже).

Провести тестирование и отладку всех модулей программы.

Оформить итоговую техническую документацию и разместить её вместе с кодом в общем репозитории.

3. Описание системы

Система предназначена для управления процессом продажи кальянов и сопутствующих услуг в баре. Основной функционал включает:

Ассортимент кальянов и вкусов: ведение списка доступных кальянов (с их характеристиками и ценами) и вкусов табака. Возможность добавлять новые позиции, редактировать или удалять имеющиеся.

Меню и оформление заказа: формирование меню текущих предложений, создание заказа из одного или нескольких кальянов с выбранными вкусами. Автоматический расчёт стоимости и оформление чека на покупку.

VIP-бронирование: система ведения бронирования столиков или закладок под заказ для VIP-клиентов. Учет VIP-статуса клиентов, приоритетное резервирование и индивидуальные условия обслуживания.

История заказов: хранение полной истории всех оформленных заказов с указанием состава заказа, стоимости, даты/времени и данных клиента (если применимо). Возможность просмотра отчёта по продажам за выбранный период.

Авторизация администратора: вход в систему под учётной записью администратора с логином и паролем. Только авторизованный администратор имеет доступ к функциям управления ассортиментом, меню и отчетности.

На текущем этапе все данные хранятся во временных структурах программы (например, списках) или в локальных файлах. База данных не задействована.

4. Этапы разработки

Консольная версия: создание базового приложения с текстовым интерфейсом. Реализуются все основные функции системы через консольные команды и диалоги. Данные хранятся в оперативной памяти и/или в файлах. Проверяется корректность логики обработки заказов и взаимодействия модулей.

Графический интерфейс (GUI): перенос функционала в графическое приложение с использованием Java (Swing или JavaFX). Интерфейс выполняется в тёмной цветовой схеме. Добавляются визуальные элементы управления (кнопки, таблицы, формы ввода) для удобной работы. Улучшается пользовательский опыт.

Интеграция базы данных: переход к хранению данных в реляционной СУБД для длительного хранения информации о кальянах, вкусах, заказах и клиентах. Этап планируется в будущем, детали проектирования базы данных будут описаны в отдельном документе.

5. Распределение задач по участникам

Проект выполняется командой из трёх разработчиков с примерно равным распределением работы. Ответственность распределена следующим образом:

Разработчик 1: модуль управления ассортиментом кальянов и вкусов, организация меню. Реализует функции добавления, изменения и удаления позиций ассортимента. Обеспечивает сохранение данных и связь с другими модулями. Проводит собственное модульное тестирование своего кода.

Разработчик 2: модуль обработки заказов и ведения истории. Реализует логику оформления заказа (выбор позиций, расчёт стоимости), VIP-бронирование и сохранение истории продаж. Обеспечивает корректное взаимодействие с модулем ассортимента. Тестирует функции своего модуля.

Разработчик 3: пользовательский интерфейс и авторизация. Создаёт консольные команды и затем GUI-экраны для работы программы, реализует систему входа администратора по логину и паролю. Интегрирует модули 1 и 2, обеспечивая их взаимодействие через интерфейс. Проводит тестирование интеграции.

Каждый разработчик тестирует свою часть программного кода отдельно. После реализации функций проводится совместное интеграционное тестирование всей системы, в ходе которого проверяется корректность работы взаимодействующих модулей.

6. План тестирования

Модульное тестирование: каждый разработчик создаёт набор юнит-тестов (например, с помощью JUnit) для проверки работы своего модуля. Тесты покрывают основные функции модуля, проверяют граничные и исключительные ситуации.

Функциональное тестирование: проверка работы системы в целом по пользовательским сценариям. Тестируются основные функции: создание заказа, расчёт чека, добавление/удаление позиций из меню, бронирование VIP, авторизация и др. Цель — убедиться, что система выполняет заявленные функции согласно ТЗ.

Интеграционное тестирование: проверка взаимодействия между модулями (ассортимент, заказы, интерфейс и т.д.). Убедиться, что модули корректно обмениваются данными (например, каталог кальянов из одного модуля доступен в другом) и общая работа системы стабильна.

7. Принципы командной работы и хранения проекта

Единый репозиторий: весь исходный код и документация проекта хранятся в одном репозитории на GitLab. Это гарантирует единый источник правды и доступность проекта всем участникам.

Ветки разработки: каждый разработчик работает в отдельной ветке для своей задачи или нового функционала. Такой подход позволяет вести параллельную разработку без конфликтов в основной ветке.

Код-ревью и слияние: после завершения работы над функцией разработчик создаёт merge request для проверки изменений. Остальные участники просматривают код и после одобрения ветки осуществляется её слияние с основной (main/master). Это обеспечивает контроль качества и согласованность изменений.

Хранение ТЗ и документации: техническое задание, документация и исходный код размещаются в репозитории (например, в корневой папке или каталоге docs/). Благодаря этому вся необходимая информация доступна прямо в проекте.

Регулярные коммиты: участники регулярно сохраняют промежуточные изменения с информативными сообщениями. Это облегчает отслеживание истории проекта и упрощает выявление ошибок.

8. Общие требования к интерфейсу и системе

Платформа и язык: приложение разрабатывается на языке Java (рекомендуется версия 8 или выше) и предназначено для запуска на персональных компьютерах (кроссплатформенность Windows/Linux).

Консольный интерфейс: на начальной фазе (этап 1) реализуется текстовый интерфейс командной строки. Пользователь взаимодействует с системой через команды и текстовые меню.

Графический интерфейс: на втором этапе разрабатывается GUI. Требуется тёмная цветовая тема оформления (тёмные фоны, светлый текст), современный минималистичный дизайн. Интерфейс должен быть интуитивно понятным и удобным для пользователя.

Хранение данных: до этапа с базой данных вся информация хранится в оперативной памяти приложения или в локальных файлах (например, текстовых или JSON). СУБД не используется на данном этапе.

Другие требования: обеспечение стабильной работы без зависаний, корректная обработка некорректного ввода пользователя (валидация данных). При необходимости допускается расширение функциональности по согласованию с руководителем проекта.
